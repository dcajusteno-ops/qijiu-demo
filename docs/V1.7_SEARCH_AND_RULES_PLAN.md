# v1.7 搜索 MVP 与自动规则引擎规划

## 概述

本文档将 v1.7 的规划范围收敛为两个功能：

1. 搜索 MVP
2. 自动规则引擎方案

本次规划的目标很明确：

- 让用户能在大量 ComfyUI 输出图片中更快找到目标图片
- 让重复性的整理动作自动执行，减少手工打标、分类和移动文件的成本

---

## 一、搜索 MVP

### 1. 目标

在图库页提供一个统一搜索入口，让用户可以通过以下信息快速定位图片：

- 文件名
- 正向 Prompt
- 反向 Prompt
- 模型
- 采样器
- LoRA
- 标签
- 笔记

### 2. 范围

本次 MVP 包含：

- 图库页顶部增加一个全局搜索框
- 输入关键词后对当前图片集合实时过滤
- 搜索结果可与现有标签筛选、智能相册筛选、排序、分页叠加使用
- 搜索结果数量正确更新
- 提供清空搜索功能
- 可选增加简单命中标识，例如：`命中 Prompt`、`命中 Model`、`命中 Tag`

本次 MVP 不包含：

- 高级搜索语法
- 模糊搜索排序
- 搜索历史
- 已保存搜索
- 全局命令面板式搜索

### 3. 可复用的现有基础

当前项目里已经具备很多可直接复用的能力：

- `desktop-source/app.go`
  - 已有图片元数据缓存
  - `ImageMetaCacheEntry.SearchText` 字段已经存在
- `desktop-source/frontend/src/composables/useImages.js`
  - 已集中管理图库筛选逻辑
- `desktop-source/frontend/src/components/ImageGallery.vue`
  - 适合放置图库级搜索框
- `desktop-source/frontend/src/components/ImageCard.vue`
  - 可用于显示搜索命中提示

### 4. 开发清单

#### 后端

- [ ] 检查 `app.go` 中 `SearchText` 的现有生成逻辑
- [ ] 补齐 `SearchText` 聚合内容，至少包含：
  - 正向 Prompt
  - 反向 Prompt
  - 模型
  - 采样器
  - LoRA
  - 文件名
- [ ] 评估标签和笔记是否放在前端聚合，还是补充到后端搜索字段中
- [ ] 确保 metadata warmup 不影响当前启动性能和刷新体验

#### 前端

- [ ] 在 `useImages.js` 中新增 `searchQuery` 状态
- [ ] 在最终图片过滤链路中接入搜索过滤
- [ ] 保证搜索与以下条件叠加时逻辑正确：
  - 当前目录
  - 标签筛选
  - 智能相册筛选
  - 日期、尺寸等现有过滤条件
- [ ] 在 `ImageGallery.vue` 顶部增加搜索输入框
- [ ] 在有搜索词时显示清空按钮
- [ ] 增加“无搜索结果”空状态提示
- [ ] 在图库头部或结果区域展示当前命中数量
- [ ] 可选：在 `ImageCard.vue` 增加轻量命中来源提示

#### 交互体验

- [ ] 占位文案建议：`搜索文件名、Prompt、模型、LoRA...`
- [ ] 搜索采用不区分大小写的包含匹配
- [ ] 如果大图库下性能有压力，再考虑输入防抖
- [ ] 同一视图内保留当前搜索词
- [ ] 仅在切换到明显不适合保留搜索词的视图时，再考虑自动清空

### 5. 推荐实现策略

建议第一版采用“后端元数据文本 + 前端补充聚合”的方式：

- 后端继续维护 metadata 相关的 `SearchText`
- 前端补充合并以下内容：
  - `img.name`
  - `img.prompt`
  - `img.model`
  - 标签文本
  - 笔记文本

这样做的好处是：

- 开发成本低
- 改动范围可控
- 不需要额外设计一套新的搜索 API

### 6. 验收标准

- 用户输入模型名后，能立即筛出对应图片
- 用户输入 Prompt 片段后，能立即筛出对应图片
- 用户输入标签名或笔记片段后，能立即筛出对应图片
- 搜索结果和当前目录导航能正确联动
- 搜索结果和智能相册筛选能正确叠加
- 分页基于过滤后的结果重新计算
- “无结果”状态不会和“当前目录本来就没有图片”混淆

### 7. 建议开发顺序

1. 先检查并补齐后端 `SearchText`
2. 在 `useImages.js` 中接入搜索状态与搜索过滤
3. 在 `ImageGallery.vue` 中加入搜索 UI
4. 补充结果计数与空状态
5. 视时间决定是否补充命中提示

## 二、自动规则引擎方案

### 1. 目标

在图片进入图库后，根据用户定义的规则自动执行整理动作，减少重复劳动。

自动规则引擎的核心价值是：

- 自动打标签
- 自动归类
- 自动加入收藏组
- 自动移动到目标目录

### 2. v1 版本定位

这个功能在 v1.7 中建议定义为“规则引擎 v1”，而不是完整自动化平台。

第一版重点放在：

- 条件清晰
- 动作安全
- 执行结果可感知
- 尽量复用现有标签、收藏、移动文件能力

### 3. 推荐首版范围

首版支持的条件：

- 模型包含某关键词
- 采样器等于某值
- LoRA 包含某值
- 尺寸等于某值
- 文件名包含某关键词
- 正向 Prompt 包含某关键词
- 反向 Prompt 包含某关键词

首版支持的动作：

- 添加标签
- 设置状态
- 加入收藏分组
- 移动到指定目录

首版暂不支持：

- 复杂嵌套规则
- 任意布尔表达式编辑器
- 定时任务
- 删除文件
- 外部 webhook
- 多层条件分支

### 4. 示例规则

- 如果模型包含 `pony`，自动添加标签 `Pony`
- 如果 LoRA 包含 `detail`，自动添加标签 `细节增强`
- 如果尺寸等于 `1024x1536`，自动移动到 `日期归档/竖图`
- 如果 Prompt 包含 `girl`，自动加入收藏组 `角色候选`
- 如果采样器等于 `Euler a`，自动设置状态为 `待筛`

### 5. 数据结构建议

建议新增文件：

- `data/auto-rules.json`

建议结构如下：

```json
[
  {
    "id": "rule-uuid",
    "name": "Pony 自动打标",
    "enabled": true,
    "matchMode": "all",
    "conditions": [
      { "field": "model", "operator": "contains", "value": "pony" }
    ],
    "actions": [
      { "type": "add_tag", "value": "Pony" }
    ],
    "createdAt": "2026-04-16T08:00:00Z",
    "updatedAt": "2026-04-16T08:00:00Z"
  }
]
```

建议枚举如下：

- `field`
  - `model`
  - `sampler`
  - `lora`
  - `dimensions`
  - `filename`
  - `prompt`
  - `negative`
- `operator`
  - `contains`
  - `equals`
  - `starts_with`
  - `ends_with`
- `action.type`
  - `add_tag`
  - `set_status`
  - `add_favorite_group`
  - `move_to_folder`

### 6. 执行时机建议

建议首版在以下时机执行规则：

- watcher 检测到新图片后
- 上传/导入图片完成后
- 用户手动刷新并发现新增图片后

推荐处理顺序：

1. 新图片被扫描到
2. 图片元数据被读取
3. 规则匹配执行
4. 对命中的图片执行动作
5. 触发界面刷新
6. 记录执行结果

推荐动作执行顺序：

1. 添加标签
2. 设置状态
3. 加入收藏组
4. 移动目录

原因是：

- 移动路径的动作应该尽量放在最后，避免前面动作仍依赖旧相对路径时出现问题

### 7. 界面方案

建议首版入口放在个人中心页面，而不是侧边栏工具区。

推荐做法：

- 在 `个人中心` 页面新增一个“自动规则”设置区块
- 在该区块内提供“规则管理”入口按钮
- 规则管理可以采用弹窗，也可以采用个人中心内嵌卡片区域

建议提供一个规则管理区域，支持：

- 查看规则列表
- 新建规则
- 编辑规则
- 启用/停用规则
- 删除规则

建议编辑器具备这些字段：

- 规则名称
- 是否启用
- 匹配模式：`全部满足` / `任一满足`
- 条件列表
- 动作列表

建议个人中心中额外提供“规则控制能力”，至少包含：

- 单条规则启用/停用
- 查看规则当前条件和动作摘要
- 编辑规则
- 删除规则
- 查看最近一次执行结果
- 查看最近一次命中数量
- 查看最近一次失败信息

如果开发时间允许，建议进一步增加：

- 调整规则执行顺序
- 批量启用/停用规则
- 只查看已启用规则
- 只查看执行失败规则

### 7.1 个人中心页面结构建议

建议在个人中心新增一个独立的“自动规则”功能区，放在现有资料设置和快捷入口区域附近，作为偏“工作流设置”的模块。

推荐拆分为 4 个部分：

#### A. 规则概览卡

作用：

- 让用户一进入个人中心就知道自动规则当前是否开启、是否正常工作

建议展示内容：

- 自动规则总数
- 已启用规则数
- 最近一次执行时间
- 最近一次命中图片数
- 最近一次失败数

建议操作：

- `打开规则管理`
- `新建规则`
- 可选：总开关 `启用自动规则`

#### B. 规则列表卡

作用：

- 展示当前已有的规则，并提供最核心的控制能力

每条规则建议展示：

- 规则名称
- 是否启用
- 条件摘要
- 动作摘要
- 最近一次命中结果
- 最近一次执行状态

每条规则建议提供的操作：

- 启用/停用
- 编辑
- 删除
- 可选：上移 / 下移

建议列表支持的过滤：

- 全部规则
- 仅看已启用
- 仅看已停用
- 仅看最近失败

#### C. 执行状态卡

作用：

- 让用户确认自动化是否真的生效，而不是“设了规则但不知道有没有跑”

建议展示内容：

- 最近一次规则执行时间
- 最近一次命中的规则名称
- 最近一次命中的图片数量
- 最近一次失败原因

如果时间允许，可扩展为：

- 最近 10 条规则执行记录
- 每条记录包含：
  - 规则名
  - 执行动作
  - 命中图片数
  - 成功/失败状态

#### D. 新建 / 编辑规则弹窗

作用：

- 承载规则的新增与修改，不建议首版直接塞进主页面，避免个人中心过重

建议字段结构：

- 规则名称
- 是否启用
- 匹配模式
  - 全部满足
  - 任一满足
- 条件列表
  - 字段
  - 运算符
  - 值
- 动作列表
  - 动作类型
  - 动作目标值

建议底部操作：

- 保存规则
- 取消
- 删除规则（编辑时显示）

### 7.2 推荐的页面布局顺序

建议个人中心中的展示顺序如下：

1. 自动规则概览卡
2. 规则列表卡
3. 执行状态卡
4. 新建 / 编辑规则弹窗

这样安排的原因：

- 用户先看到“当前状态”
- 再看到“有哪些规则”
- 再看到“最近运行结果”
- 最后通过弹窗进入详细编辑

### 7.3 前端拆分建议

建议前端不要把所有逻辑直接堆进 `ProfileCenter.vue`，而是采用“个人中心承载入口，规则模块独立组件”的方式。

推荐拆分：

- `ProfileCenter.vue`
  - 放自动规则总入口和概览信息
- `AutoRulesPanel.vue`（建议新增）
  - 放规则列表与规则状态区域
- `AutoRulesDialog.vue`
  - 放新建 / 编辑规则表单
- `AutoRuleItem.vue`（可选新增）
  - 放单条规则卡片
- `AutoRuleRunLog.vue`（后续可选）
  - 放规则执行记录

### 7.4 推荐的首版最小可见结构

如果 v1.7 只做最小可用版本，建议至少保证个人中心里有这三块：

- 自动规则概览
- 规则列表
- 新建/编辑规则弹窗

执行状态卡可以先做成简单摘要，不一定一开始就做完整日志。

### 7.5 简约风布局要求

自动规则区域建议采用简约风，不要花里胡哨，不要做成“控制台大屏”或“炫技面板”。

界面风格要求：

- 以信息清晰、操作直接为优先
- 减少装饰性插画、渐变光效、悬浮发光、夸张阴影
- 减少大面积彩色块，尽量使用当前站内已有的卡片和边框体系
- 控制视觉层级，避免一个页面上出现太多强调色
- 动效只保留基础开合、悬停、禁用态，不做复杂动画

布局原则：

- 一屏先看到核心状态和主要操作
- 单条规则信息保持短而清楚，不做大段说明文案
- 列表优先于瀑布流，方便扫读和批量管理
- 编辑规则时使用整洁表单，不堆太多提示块
- 每个卡片只承担一个主要职责，避免功能混杂

不建议出现的设计：

- 大面积渐变背景
- 强烈玻璃拟态
- 过多图标堆叠
- 彩色徽章满屏都是
- 每个按钮都做高亮强调
- 重复出现说明文字

## 二点一、自动规则 UI 文案与简约布局清单

本节用于指导“个人中心中的自动规则区域”如何以简约风形式落地。

### 1. 页面布局建议

推荐采用纵向分区布局，保持清爽：

1. 自动规则概览卡
2. 规则列表卡
3. 执行状态摘要卡

弹窗独立处理：

- 新建规则弹窗
- 编辑规则弹窗

推荐布局特点：

- 整体左对齐
- 卡片间距统一
- 标题简短
- 每块卡片控制在 1 个主要目标内

### 2. 自动规则概览卡

定位：

- 放在个人中心中偏上位置
- 作为自动规则模块入口

建议展示字段：

- 标题：`自动规则`
- 描述：`把重复整理动作交给系统自动完成。`
- 统计项：
  - `规则总数`
  - `已启用`
  - `最近执行`
  - `最近命中`

建议按钮：

- 主按钮：`新建规则`
- 次按钮：`管理规则`

如果做总开关，建议文案：

- 开关标题：`启用自动规则`
- 说明文案：`关闭后不会自动处理新进入图库的图片。`

### 3. 规则列表卡

定位：

- 放在概览卡下方
- 作为主要操作区

卡片标题建议：

- 标题：`规则列表`
- 描述：`查看、启用、编辑和删除已有规则。`

列表顶部建议控件：

- 搜索框占位：`搜索规则名称`
- 筛选项：
  - `全部`
  - `已启用`
  - `已停用`
  - `最近失败`

单条规则建议采用简洁横向列表项，不建议做复杂大卡片。

每条规则建议显示：

- 规则名称
- 启用状态
- 条件摘要
- 动作摘要
- 最近一次执行结果

每条规则建议按钮：

- `启用` / `停用`
- `编辑`
- `删除`

如果支持顺序调整，按钮建议尽量弱化：

- `上移`
- `下移`

### 4. 执行状态摘要卡

定位：

- 放在规则列表下方
- 用于建立用户对自动化的信任感

建议标题：

- 标题：`执行状态`
- 描述：`查看最近一次自动处理是否正常运行。`

建议字段：

- `最近执行时间`
- `最近命中规则`
- `最近处理图片数`
- `最近失败原因`

没有数据时建议文案：

- `还没有执行记录，规则会在新图片进入图库后自动运行。`

### 5. 新建 / 编辑规则弹窗

建议整体风格：

- 标题明确
- 表单分组清楚
- 不要堆太多说明文案
- 默认显示最常用字段

弹窗标题建议：

- 新建时：`新建规则`
- 编辑时：`编辑规则`

字段建议：

- `规则名称`
- `是否启用`
- `匹配模式`
- `条件`
- `动作`

匹配模式文案建议：

- `全部满足`
- `任一满足`

条件区建议结构：

- 字段选择
- 运算符选择
- 值输入

动作区建议结构：

- 动作类型
- 动作目标值

按钮建议：

- 主按钮：`保存规则`
- 次按钮：`取消`
- 危险按钮：`删除规则`（仅编辑态显示）

### 6. 建议使用的简约文案

推荐使用短句，不要过度解释。

适合的文案风格：

- `当图片满足这些条件时，自动执行整理动作。`
- `规则会在新图片进入图库后自动运行。`
- `你可以随时停用某条规则。`
- `最近一次执行没有发现异常。`
- `这条规则暂时未启用。`

不建议的文案风格：

- 过于技术化
- 一段话解释多个概念
- 强烈营销式表达
- 太多感叹号或情绪化措辞

### 7. 首版最小 UI 清单

如果要保持简约并控制开发成本，建议首版 UI 只做这些：

- 一个自动规则概览卡
- 一个规则列表卡
- 一个简单执行状态卡
- 一个新建 / 编辑规则弹窗

首版可以暂时不做：

- 复杂图表
- 规则命中趋势图
- 彩色状态矩阵
- 多层折叠规则树
- 富文本提示面板

### 8. 推荐的实现原则

为了保持简约风，开发时建议遵守以下原则：

- 优先复用现有 `Card`、`Button`、`Input`、`Badge` 组件
- 使用统一圆角、边框、留白，不单独发明一套视觉语言
- 状态颜色只在必要时使用，例如：
  - 成功
  - 失败
  - 已停用
- 默认状态尽量使用中性色
- 一条规则的信息不要超过两行摘要

## 二点二、个人中心自动规则区块静态线框说明

本节使用纯文字说明“个人中心页面中的自动规则区域”应如何排版，目标是让页面结构在没有正式设计稿的情况下也能直接进入实现。

### 1. 页面中的相对位置

建议把“自动规则”放在个人中心页面的中下部，靠近“设置”和“快捷入口”这一类偏管理型内容，而不要放在最顶部抢占个人资料主视觉区域。

推荐顺序如下：

1. 个人资料主卡
2. 图片卡片
3. 基础设置或偏好设置
4. 自动规则区块
5. 快捷入口

这样安排的原因：

- 用户进入个人中心时，先看到“我”的信息
- 然后看到常用设置
- 自动规则作为“工作流设置”自然出现
- 不会破坏个人中心当前较轻的页面节奏

### 2. 自动规则区块整体结构

推荐采用纵向三段式结构：

```text
+------------------------------------------------------+
| 自动规则概览卡                                       |
| 标题 / 简述 / 统计 / 主按钮 / 次按钮 / 总开关        |
+------------------------------------------------------+

+------------------------------------------------------+
| 规则列表卡                                           |
| 顶部工具条                                           |
| 搜索框 | 筛选项 | 新建规则                           |
| ---------------------------------------------------- |
| 规则项 1                                             |
| 规则项 2                                             |
| 规则项 3                                             |
+------------------------------------------------------+

+------------------------------------------------------+
| 执行状态摘要卡                                       |
| 最近执行 / 最近命中 / 最近失败 / 简短说明            |
+------------------------------------------------------+
```

首版不建议做左右复杂分栏，也不建议把规则列表和执行状态做成过多嵌套区域。

### 3. 自动规则概览卡线框

建议结构如下：

```text
+------------------------------------------------------+
| 自动规则                                    [总开关] |
| 把重复整理动作交给系统自动完成。                     |
|                                                      |
| 规则总数   已启用   最近执行   最近命中              |
|  6         4        10 分钟前  12 张                 |
|                                                      |
| [新建规则]   [管理规则]                              |
+------------------------------------------------------+
```

布局说明：

- 第一行左侧放标题，右侧放总开关
- 第二行是一句简短说明
- 第三行是 4 个统计项，尽量横向排布
- 底部放 1 个主按钮和 1 个次按钮

视觉建议：

- 标题正常加粗即可
- 统计值略大于标签文字
- 不需要大图标，也不需要装饰图形

### 4. 规则列表卡线框

建议结构如下：

```text
+------------------------------------------------------+
| 规则列表                                             |
| 查看、启用、编辑和删除已有规则。                     |
|                                                      |
| [搜索规则名称........] [全部 v]         [新建规则]   |
|------------------------------------------------------|
| Pony 自动打标                           [开关]       |
| 条件：模型包含 pony                                   |
| 动作：添加标签 Pony                                   |
| 最近执行：成功，命中 8 张            [编辑] [删除]   |
|------------------------------------------------------|
| 竖图自动归档                           [开关]       |
| 条件：尺寸等于 1024x1536                              |
| 动作：移动到 日期归档/竖图                            |
| 最近执行：成功，命中 3 张            [编辑] [删除]   |
|------------------------------------------------------|
| Euler a 待筛规则                        [开关]       |
| 条件：采样器等于 Euler a                              |
| 动作：设置状态 待筛                                   |
| 最近执行：未运行                    [编辑] [删除]   |
+------------------------------------------------------+
```

布局说明：

- 顶部两行是标题和描述
- 第三行是工具条：搜索、筛选、新建规则
- 下方是规则项列表
- 每个规则项高度尽量控制在 3 到 4 行信息

规则项建议信息层级：

- 第一层：规则名称 + 启用开关
- 第二层：条件摘要
- 第三层：动作摘要
- 第四层：最近执行结果 + 操作按钮

首版不建议在单条规则上展示太多内容，例如：

- 复杂的条件树
- 多行日志
- 大量彩色状态标签

### 5. 执行状态摘要卡线框

建议结构如下：

```text
+------------------------------------------------------+
| 执行状态                                             |
| 查看最近一次自动处理是否正常运行。                   |
|                                                      |
| 最近执行时间：2026-04-16 10:32                       |
| 最近命中规则：Pony 自动打标                          |
| 最近处理图片：12 张                                  |
| 最近失败原因：无                                     |
+------------------------------------------------------+
```

布局说明：

- 保持单列信息展示
- 每行一个字段
- 不做复杂表格

如果没有数据：

```text
+------------------------------------------------------+
| 执行状态                                             |
| 还没有执行记录，规则会在新图片进入图库后自动运行。   |
+------------------------------------------------------+
```

### 6. 新建 / 编辑规则弹窗线框

建议弹窗结构如下：

```text
+------------------------------------------------------+
| 新建规则                                      [X]    |
| 为满足条件的图片自动执行整理动作。                   |
|------------------------------------------------------|
| 规则名称 [______________________________]            |
| 是否启用 [开关]                                      |
| 匹配模式 [全部满足 v]                                |
|                                                      |
| 条件                                                 |
| [字段 v] [运算符 v] [值_____________] [删除]         |
| [新增条件]                                           |
|                                                      |
| 动作                                                 |
| [动作类型 v] [目标值_____________] [删除]            |
| [新增动作]                                           |
|------------------------------------------------------|
|                              [取消] [保存规则]      |
+------------------------------------------------------+
```

编辑态可在左下或底部补一个弱化的危险按钮：

```text
[删除规则]
```

布局说明：

- 表单从上到下线性排列
- 条件和动作都使用重复行结构
- 不要一开始就塞太多帮助说明
- 对于字段解释，优先使用 placeholder 或简短说明，而不是大段文字

### 7. 移动端或窄宽度时的降级方式

如果个人中心在窄宽布局下显示，建议自动规则区域按以下方式降级：

- 概览卡中的统计项改为 2 列换行
- 规则列表工具条改为两行
  - 第一行：搜索框
  - 第二行：筛选项 + 新建按钮
- 单条规则项的右侧操作改为折行
- 弹窗宽度收窄，但字段顺序不变

不建议在窄屏下改成完全不同的信息架构，尽量保持一致性。

### 8. 首版最小实现线框

如果需要进一步压缩首版工作量，建议优先做到下面这个最小版本：

```text
+------------------------------------------------------+
| 自动规则                                             |
| 规则总数  已启用  最近执行                           |
| [新建规则]                                           |
+------------------------------------------------------+

+------------------------------------------------------+
| 规则列表                                             |
| [搜索规则名称........]                 [全部 v]      |
|------------------------------------------------------|
| 规则名称 + 启用开关                                  |
| 条件摘要                                             |
| 动作摘要                                             |
| [编辑] [删除]                                        |
+------------------------------------------------------+
```

这个版本已经能满足：

- 看状态
- 看规则
- 新建规则
- 编辑规则
- 启用停用规则

也是最符合“简约风、先做实用性”的落地方式。

### 9. 安全要求

自动规则引擎首版必须保守设计：

- [ ] 不提供删除文件动作
- [ ] 不允许跨受管目录范围写入
- [ ] 移动目录时避免覆盖冲突
- [ ] 单张图片执行失败不影响整体刷新流程
- [ ] 防止移动后再次触发自身形成循环
- [ ] 所有动作都只作用于受管图片路径

### 10. 可见性与反馈建议

自动化功能如果没有反馈，用户很难建立信任，因此建议首版至少提供：

- 最近一次执行时间
- 最近一次命中数量
- 最近一次失败信息

后续可以再扩展：

- 最近规则执行日志面板
- 每条规则的命中统计

这些信息建议直接放在个人中心的“自动规则”区域中，避免用户还要跳到其他页面才能查看控制结果。

### 11. 验收标准

- 用户可以创建一条包含“一个条件 + 一个动作”的规则
- 新扫描进来的匹配图片会自动执行相应动作
- 规则可以启用和停用，不需要删除才能失效
- 单条规则出错不会导致整个图库刷新失败
- 用户能感知规则是否执行过、执行了多少张、是否有错误
- 移动目录动作必须经过路径安全校验
